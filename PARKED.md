## Cleared

- Browser panel width "defect" — **not a defect** (round 46). The panel is 320 physical px as designed; the 182 px reading came from a DPI-unaware PowerShell probe, which Windows virtualizes to logical pixels (320 / 1.75 = 182). Measure DPI-scaled geometry from a DPI-aware process, or expect scaled numbers.

- Embedded browser panel (M12 stage 2) — **working, proven live 2026-07-22 (round 45)**. Root cause of round 44's invisible panel: WebView2's controller must live on the UI thread that owns its parent `HWND`, and it creates its own child windows. Round 44 instead made a child window on a *separate* browser thread, which never took `WS_VISIBLE`. The fix hands `CreateCoreWebView2Controller` the winit `HWND` directly from the winit thread and creates no window of its own; creation is fire-and-forget (handlers install the controller, and bounds/visibility/URL requested beforehand are replayed) so the event loop never blocks. Verified by `EnumChildWindows`: `Chrome_WidgetWin_0` → `Chrome_WidgetWin_1` → `Chrome_RenderWidgetHostHWND`, all parented under the gmux window, all `visible=True`, and the render widget titled **"Example Domain"** — i.e. example.com actually loaded and painting inside the terminal window.

- WebView2 browser pane "worked once, then silently did nothing" — root cause found and fixed round 35. Two bugs: (1) once the user closed the browser window, `navigate` kept posting to a destroyed HWND — a silent no-op forever, which is exactly the reported symptom; `navigate` now returns false when the window is gone and the GUI opens a fresh pane. (2) a URL arriving while WebView2 was still initializing was drained against a `None` controller and dropped; it is now held as `pending` and navigated when the controller lands. Proven live: browse → window opens; close it; browse again → new window opens (this second step failed before the fix).
